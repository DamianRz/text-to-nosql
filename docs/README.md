# Documentation

## Purpose

Keep the real architecture, operational boundaries, and improvement roadmap in one place so knowledge does not drift between `README`, tests, and implementation files.

## Index

- `architecture.md`: technical map and end-to-end flow
- `improvement-plan.md`: visual and code improvement roadmap

## Recommended usage

1. Read `architecture.md` before changing the pipeline, API routes, or primary UI.
2. Read `improvement-plan.md` before opening refactor or redesign work.
3. Keep `README.md` as the quick-start entry point and `docs/` as the operational reference.
